# راهنمای ماژول پایگاه داده و ذخیره‌سازی (Module Database)
**مسیر در پروژه:** `docs/04_Modules_Documentation/module_database.md`
**هدف:** مدیریت یکپارچه و امن اتصالات پایگاه داده (SQLite)، اجرای کوئری‌ها، و حفاظت از مرزهای سیستم با استفاده از اشیاء انتقال داده (DTO).

این سند راهنمای توسعه‌دهنده برای فایل‌های موجود در دایرکتوری `src/database/` است.

---

## ۱. فلسفه طراحی: دیتابیس فقط یک «جزئیات» است

در معماری پورت‌ها و آداپتورها (Hexagonal Architecture)، پایگاه داده قلب نرم‌افزار نیست؛ بلکه صرفاً یک «ابزار جانبی» (Detail) یا یک «آداپتور» است که در بیرونی‌ترین لایه سیستم قرار دارد.

موتور اصلی بازی (Core) و منطق ریاضی (Mechanics) هیچ اهمیتی نمی‌دهند که داده‌ها در فایل متنی ذخیره می‌شوند، در SQLite هستند یا در یک سرور ابری. آن‌ها فقط به یک واسط (Interface) دستور می‌دهند. وظیفه شما در این ماژول، پنهان کردن تمام پیچیدگی‌های کار با SQLite در پشت پرده است، تا بقیه تیم هرگز درگیر دستورات SQL نشوند.

---

## ۲. ساختار و وظایف زیرماژول‌ها

برای جلوگیری از درهم‌تنیدگی کدهای اتصال، مدل‌ها و کوئری‌ها، دایرکتوری پایگاه داده به فایل‌های کاملاً تخصصی تقسیم شده است:

### 🔌 الف) مدیریت اتصال (`connection_manager.py`)
این فایل منحصراً وظیفه باز کردن و بستن ایمن ارتباط با فایل پایگاه داده (`game_data.db`) را بر عهده دارد. استفاده از این کلاس از قفل شدن (Lock) دیتابیس جلوگیری می‌کند.

### 🗄️ ب) مدل‌های داخلی (`models.py`)
اینجا ساختار جداول دیتابیس به صورت کلاس تعریف می‌شود. این کلاس‌ها فقط برای استفاده داخلی خودِ دیتابیس هستند و **هرگز** نباید از این پوشه خارج شوند.

### 🛡️ ج) اشیاء انتقال داده (`dtos.py`)
مهم‌ترین مفهوم امنیتی در مرزهای سیستم. DTO (Data Transfer Object) یک کلاس ساده و بدون منطق است که به عنوان "پاکت نامه" عمل می‌کند. دیتای دیتابیس درون این پاکت قرار گرفته و سپس برای نمایش به لایه UI ارسال می‌شود.

### 📜 د) مخزن دستورات خام (`queries.py`)
تمام دستورات SQL (مثل `SELECT`، `INSERT` و `CREATE TABLE`) باید به صورت متغیرهای متنی (String) در این فایل ذخیره شوند. نوشتن دستور خام SQL در لابه‌لای کدهای اجرایی ممنوع است.

### 🛠️ هـ) آداپتور اصلی (`sqlite_repository.py`)
این فایل مهم‌ترین قطعه اجرایی شماست. کلاس `SQLiteRepository` باید اینترفیسِ `i_database.py` (تعریف شده در پوشه `contracts`) را به طور کامل پیاده‌سازی کند.

## ۳. الگوی اشیاء انتقال داده (DTO Pattern)

چرا نمی‌توانیم مستقیماً دیتای خوانده شده از دیتابیس را به لایه UI بدهیم؟
فرض کنید جدول شما دارای ستون‌های `id`، `player_name`، `score`، `mac_address` و `created_at` است. اگر این ردیف را مستقیماً به UI بدهید (نشت داده)، علاوه بر مصرف حافظه اضافی، ممکن است اطلاعات حساسی به اشتباه نمایش داده شود. 

شما باید یک DTO بسازید که فقط فیلدهای مورد نیاز را دارد:
```python
# در فایل dtos.py
from dataclasses import dataclass

@dataclass(frozen=True)
class LeaderboardEntryDTO:
    """این شیء فقط شامل دیتای ایمن برای نمایش است و قابل تغییر نیست"""
    rank: int
    player_name: str
    score: int
````

عملیات نگاشت (Mapping) در ریپازیتوری اتفاق می‌افتد: دیتای خام از SQLite خوانده می‌شود، تبدیل به `LeaderboardEntryDTO` می‌شود و در نهایت این DTO به لایه UI ارسال می‌گردد.

## ۴. خطوط قرمز معماری (Strict Constraints)

توسعه‌دهنده لایه پایگاه داده موظف است این قوانین را بدون هیچ‌گونه اغماضی رعایت کند:

۱. **مدیریت ایمن منابع با Context Manager:**

هرگز اتصالات دیتابیس را به صورت دستی باز رها نکنید. تمام اتصالات و اجرای کوئری‌ها باید با استفاده از بلوک `with` پایتون انجام شوند تا در صورت بروز خطا، اتصال به صورت خودکار بسته شود.

- ❌ **رویکرد غلط و خطرناک:**
```Python
    conn = sqlite3.connect('game.db')
    conn.execute("...")
    # اگر اینجا برنامه کرش کند، دیتابیس برای همیشه قفل (Locked) می‌ماند!
    conn.close() 
```

- ✅ **رویکرد مهندسی:**

```Python
    with sqlite3.connect('game.db') as conn:
        conn.execute("...")
    # اتصال در هر شرایطی به صورت خودکار بسته می‌شود.
```

۲. **پیشگیری از حملات تزریق (SQL Injection):**

هرگز از قابلیت‌های قالب‌بندی رشته (مثل `f-string` یا `+`) برای چسباندن نام بازیکن یا متغیرها به دستورات SQL استفاده نکنید. **همیشه** از پارامترهای ایمن (`?`) استفاده کنید.

- ❌ **رویکرد غلط:** `execute(f"SELECT * FROM scores WHERE name = '{player_name}'")`

- ✅ **رویکرد مهندسی:** `execute("SELECT * FROM scores WHERE name = ?", (player_name,))`

۳. **منطق تجاری ممنوع (No Business Logic):**

لایه دیتابیس نباید تصمیم بگیرد که آیا امتیاز بازیکن به اندازه کافی بالا هست که در لیدربورد ذخیره شود یا نه. دیتابیس فقط یک ماشین ذخیره‌سازی احمق (Dumb Storage) است. لایه مکانیک‌ها تصمیم می‌گیرد، و فقط به دیتابیس دستور `SAVE` می‌دهد.

۴. **پایبندی بی‌نقص به قرارداد (Contract Adherence):**

کلاس `SQLiteRepository` موظف است دقیقاً تمام توابع تعریف شده در `IDatabase` را پیاده‌سازی کند. نوع ورودی‌ها (Type Hints) و خروجی‌ها باید موبه‌مو با قرارداد لایه هسته تطابق داشته باشد.
