# راهنمای ماژول ابزارها و دغدغه‌های فراگیر (Module Utils)
**هدف:** مدیریت عملیات مشترک سیستم (مانند بارگذاری فایل‌ها و ثبت وقایع) که در تمام لایه‌ها کاربرد دارند، اما به هیچ منطق خاصی از بازی وابسته نیستند.

این سند راهنمای توسعه‌دهنده برای فایل‌های موجود در دایرکتوری `src/utils/` است.

---

## ۱. فلسفه طراحی: دغدغه‌های فراگیر (Cross-cutting Concerns)

در مهندسی نرم‌افزار، مفهومی به نام «دغدغه‌های فراگیر» وجود دارد. فرض کنید می‌خواهید هر زمان که خطایی رخ داد، آن را لاگ (ثبت) کنید. این لاگ کردن، هم در لایه دیتابیس نیاز است، هم در UI و هم در مکانیک‌ها. 
بنابراین نمی‌توانید کدهای لاگ را فقط در یک پوشه خاص بنویسید. ماژول `utils` دقیقا برای همین ابزارهای عمومی ساخته شده است که همه لایه‌ها به آن‌ها نیاز دارند.

⚠️ **هشدار معماری:** پوشه `utils` نباید تبدیل به "سطل زباله" پروژه شود! کدی که جای مشخصی برایش پیدا نمی‌کنید را نباید به صورت پیش‌فرض در اینجا بیندازید. توابع این پوشه باید کاملاً عمومی (Generic) باشند.

---

## ۲. ساختار و وظایف زیرماژول‌ها

مالک این لایه وظیفه دارد دو سیستم بسیار مهم و پایه‌ای را برای استفاده سایر اعضای تیم توسعه دهد:

### 📁 الف) سیستم بارگذاری منابع (`resource_loader.py`)
لود کردن عکس‌ها (`.png`)، فونت‌ها (`.ttf`) و صداها در پایتون مستعد خطاهای بسیاری است (مانند پیدا نشدن فایل یا تفاوت آدرس‌دهی در ویندوز و لینوکس). 

* **وظایف اصلی این کلاس:**
  ۱. **مدیریت مسیرها (Path Management):** همیشه از کتابخانه `pathlib` یا `os.path` استفاده کنید تا اسلش‌ها (`/` و `\`) در سیستم‌عامل‌های مختلف به درستی مدیریت شوند. ترکیب کردن آدرس‌ها با رشته متنی ساده (مثل `assets + "/" + image.png`) ممنوع است.
  ۲. **حافظه نهان (Caching / Memoization):** اگر لایه UI صد بار دستور لود کردن `target.png` را صادر کرد، این کلاس نباید صد بار هارد دیسک را درگیر کند. باید عکس را در دفعه اول در یک دیکشنری ذخیره (Cache) کرده و در دفعات بعدی همان عکس آماده را از حافظه RAM برگرداند.

**نمونه رویکرد مهندسی برای Cache:**
```python
import os
import pygame

class ResourceLoader:
    _image_cache = {}

    @classmethod
    def load_image(cls, file_name: str) -> pygame.Surface:
        if file_name in cls._image_cache:
            return cls._image_cache[file_name] # خواندن سریع از RAM
            
        path = os.path.join("assets", "images", file_name)
        if not os.path.exists(path):
            raise FileNotFoundError(f"Missing asset: {path}")
            
        image = pygame.image.load(path).convert_alpha()
        cls._image_cache[file_name] = image # ذخیره برای دفعات بعدی
        return image
````

### 📝 ب) سیستم ثبت وقایع (`exception_logger.py`)

استفاده از دستور `print()` برای خطایابی، نشانه بارز یک برنامه‌نویس آماتور است. `print` قابل فیلتر کردن نیست و نمی‌توان خروجی آن را به راحتی در یک فایل متنی ذخیره کرد.

- **وظایف اصلی این کلاس:**
    
    شما باید یک روکش (Wrapper) روی کتابخانه استاندارد `logging` پایتون بسازید که پیام‌ها را با سطوح مختلف (Severity Levels) دسته‌بندی کند:
    
    - `INFO`: برای اطلاعات عادی (مثلاً: "دیتابیس متصل شد").
    - `WARNING`: برای خطاهای غیربحرانی (مثلاً: "فریم‌ریت سیستم افت کرد").
    - `ERROR`: برای خطاهای بحرانی (مثلاً: "فایل تنظیمات پیدا نشد").
    - 
    - **خروجی مضاعف:** این لاگر باید همزمان خطاها را هم در ترمینال (با رنگ‌های مختلف) نشان دهد و هم در یک فایل به نام `game_errors.log` ذخیره کند.
        

### 🎵 ج) سیستم مدیریت صوتی (`audio_manager.py`)
این کلاس موظف است قرارداد `IAudioPlayer` (تعریف شده در پوشه `contracts`) را پیاده‌سازی کند و به عنوان آداپتور صوتی سیستم عمل نماید. 

* **وظایف اصلی این کلاس:**
  ۱. **راه‌اندازی امن میکسر:** در متد سازنده، باید `pygame.mixer.init()` فراخوانی شود. اگر کارت صدایی در سیستم کاربر یافت نشد، برنامه نباید کرش کند؛ بلکه باید با استفاده از `exception_logger` یک خطای `WARNING` ثبت شده و بازی به صورت بی‌صدا (Muted) ادامه یابد.
  ۲. **حافظه نهان صداها (Sound Cache):** دقیقاً مشابه لودر تصاویر، فایل‌های صوتی (`.wav` یا `.ogg`) باید فقط یک‌بار از روی هارد دیسک خوانده شده و درون یک دیکشنری ذخیره شوند (مثلاً `_sounds_cache["shoot"] = pygame.mixer.Sound(...)`). خواندن مکرر فایل صوتی از هارد در حین اجرای بازی باعث افت شدید فریم‌ریت (Lag) می‌شود.
  ۳. **پخش همزمان (Channel Management):** از آنجا که بازی دونفره است، باید تنظیماتی اعمال شود که اگر دو بازیکن به صورت کاملاً همزمان شلیک کردند، صدای شلیک اول قطع نشود. این کار با تخصیص هوشمند صداها به کانال‌های مستقل پای‌گیم (`pygame.mixer.find_channel()`) مدیریت می‌شود.
## ۳. خطوط قرمز معماری (Strict Constraints)

برای حفظ استقلال لایه ابزارها، رعایت قوانین زیر الزامی است:

۱. **کوری دامنه (Domain Blindness):**

ماژول `utils` نباید هیچ چیزی درباره منطق بازی بداند. توابع این پوشه **هرگز** نباید نامی از `Player`، `Target` یا `Combo` ببرند. اگر در `resource_loader.py` تابعی به نام `load_player_weapon()` دارید، معماری را نقض کرده‌اید. نام تابع باید کاملاً عمومی و به شکل `load_image(filename)` باشد.

۲. **ممنوعیت واردات حلقوی (No Circular Imports):**
پوشه `utils` در پایین‌ترین سطح گراف وابستگی قرار دارد. این یعنی فایل‌های `utils` **تحت هیچ شرایطی** حق ندارند فایلی از پوشه‌های `core`، `mechanics` یا `ui` را ایمپورت کنند. وارد کردن لایه‌های بالاتر به داخل لایه‌های پایین‌تر باعث خطای مرگبار `Circular Import` در پایتون می‌شود که کل برنامه را از کار می‌اندازد.
*(تنها استثنای مجاز: فایل `audio_manager.py` به دلیل تبعیت از معماری پورت و آداپتور، منحصراً اجازه دارد اینترفیس `i_audio_player` را از پوشه `contracts` ایمپورت کند؛ زیرا قراردادها در بالاترین سطح انتزاع هستند و باعث واردات حلقوی نمی‌شوند).*

۳. **توابع بدون وضعیت (Stateless Functions):**

به جز دیکشنریِ مربوط به کَش فایل‌ها (Cache)، این ماژول نباید هیچ وضعیت (State) پویایی را در خود نگه دارد. توابع این پوشه باید مانند جعبه‌ابزار عمل کنند: ورودی می‌گیرند، کار را انجام می‌دهند و خروجی را برمی‌گردانند، بدون اینکه تغییری در رفتار کلی سیستم ایجاد کنند.
