# راهنمای ماژول رابط کاربری و رندرینگ (Module UI)
**هدف:** مدیریت گرافیک، رسم اشکال، نمایش متون و ایجاد یک لایه بصری کاملاً ایزوله که هیچ اطلاعی از منطق و قوانین بازی ندارد.

این سند راهنمای توسعه‌دهنده برای فایل‌های موجود در دایرکتوری `src/ui/` است.

---

## ۱. فلسفه طراحی: رندرینگ «احمقانه» (The "Dumb" Renderer)

در معماری پورت‌ها و آداپتورها، رابط کاربری صرفاً یک «آداپتور خروجی» است. بهترین توصیف برای یک لایه UI استاندارد، اصطلاح **«رندرینگ احمقانه» (Dumb Rendering)** است. 

این یعنی لایه UI نباید هیچ‌گونه قدرت تفکر، محاسبه ریاضی یا تصمیم‌گیری داشته باشد. رابط کاربری مانند صفحه تلویزیون است؛ تلویزیون تصمیم نمی‌گیرد که نتیجه مسابقه فوتبال چند-چند شود، بلکه فقط سیگنالی که به آن می‌رسد را روی پیکسل‌ها نمایش می‌دهد.
در پروژه ما، لایه UI وضعیت (State) سیستم را از هسته بازی دریافت می‌کند و فقط وظیفه دارد آن اعداد و متغیرها را به شکل‌ها و رنگ‌های گرافیکی تبدیل کند.

---

## ۲. ساختار و وظایف زیرماژول‌ها

دایرکتوری `ui` به سه بخش منطقی تفکیک شده است تا از ایجاد کدهای تکراری (Spaghetti Code) جلوگیری شود:

### 🧩 الف) کامپوننت‌های پایه (`components/`)
این پوشه شامل آجرهای سازنده رابط کاربری است. این کلاس‌ها در سراسر پروژه بارها استفاده می‌شوند (Reusability).
* **`text_renderer.py`:** یک کلاس کمکی برای لود کردن فونت‌ها و رسم متن با سایه (Shadow) یا حاشیه (Outline) در یک مختصات خاص.
* **`button.py`:** مدیریت دکمه‌های قابل کلیک. این کلاس مختصات خود را دریافت می‌کند و اگر متد `is_clicked()` آن فراخوانی شود، خروجی `True` برمی‌گرداند.
* **`floating_number.py`:** کلاسی برای نمایش انیمیشن محوشونده اعداد (Fade-out). مثلاً وقتی بازیکن یک سیبل را می‌زند، عدد `+10` برای یک ثانیه به سمت بالا حرکت کرده و ناپدید می‌شود.

### 🎛️ ب) صفحه نمایش هدآپ (`hud/`)
این بخش وظیفه نمایش اطلاعات زنده در حین وضعیت `PLAYING` را دارد.
* **`player_dashboard.py`:** این کلاس یک متد `draw_hud(player_state)` دارد. اطلاعاتی مانند `ammo` (تیرهای باقی‌مانده)، `score` (امتیاز فعلی) و `time` (زمان باقیمانده) را از هسته گرفته و در حاشیه صفحه رسم می‌کند.

### 🖼️ ج) صفحات کلان (`screens/`)
این پوشه نماهای تمام‌صفحه (Full-Screen Views) را مدیریت می‌کند که در وضعیت‌های مختلف ماشین وضعیت (FSM) فعال می‌شوند.
* **`start_menu_screen.py`:** رسم لوگوی بازی، پس‌زمینه و نمایش فیلد ورود نام (Text Input).
* **`leaderboard_screen.py`:** دریافت لیستی از اشیاء انتقال داده (DTOs) از هسته بازی و رسم یک جدول مرتب و گرافیکی از نفرات برتر.

## ۳. پیاده‌سازی قرارداد (Implementing the Contract)

همان‌طور که در `module_contracts.md` مشخص شده، موتور اصلی بازی فقط با اینترفیس `i_renderable.py` صحبت می‌کند. توسعه‌دهنده لایه UI موظف است یک کلاس اصلی (مثلاً به نام `PygameRenderer`) بسازد که تمام متدهای آن قرارداد را با استفاده از دستورات کتابخانه `pygame` پیاده‌سازی کند.

**نمونه‌ای از پیاده‌سازی اصولی:**
```python
from contracts.i_renderable import IRenderable
import pygame

class PygameRenderer(IRenderable):
    def __init__(self, screen_surface):
        self.screen = screen_surface
        
    def clear_screen(self, color: tuple) -> None:
        # پیاده‌سازی متد تعریف شده در قرارداد
        self.screen.fill(color)
        
    def draw_entity(self, x: float, y: float, radius: float, color: tuple) -> None:
        # تبدیل مختصات ریاضی به پیکسل و رسم روی مانیتور
        pygame.draw.circle(self.screen, color, (int(x), int(y)), int(radius))
````

## ۴. خطوط قرمز معماری (Strict Constraints)

برای جلوگیری از فروپاشی معماری و حفظ یکپارچگی کدها، مالک لایه UI باید قوانین زیر را به صورت سخت‌گیرانه رعایت کند:

۱. **ممنوعیت مطلق تغییر وضعیت (Read-Only State - اصل CQS):**

لایه UI حق ندارد هیچ متغیری از وضعیت بازی را تغییر دهد. اگر کلاسی در پوشه `ui` کدی شبیه به `player.ammo -= 1` یا `target.x += 10` داشته باشد، معماری نقض شده است. UI فقط مجاز به خواندن (Read) اطلاعات است.

۲. **مدیریت ایمن منابع (Asset Lifecycle):**

فایل‌های سنگین مانند تصاویر (`.png`) و فونت‌ها (`.ttf`) **هرگز نباید** درون حلقه اصلی بازی (فاز رندر) لود شوند. عملیات `pygame.image.load` باید فقط یک‌بار در متد سازنده (`__init__`) کلاس‌ها انجام شود. لود کردن مجدد منابع در هر فریم باعث افت شدید فریم‌ریت (FPS Drop) و نشت حافظه می‌شود.

۳. **منطق تجاری ممنوع (Zero Business Logic):**

لایه رابط کاربری نباید تصمیم بگیرد چه اتفاقی در بازی می‌افتد.
- ❌ **رویکرد غلط:** بررسی اینکه آیا جان بازیکن صفر شده است تا صفحه "Game Over" رسم شود.
- ✅ **رویکرد مهندسی:** هسته بازی ماشین وضعیت را به `GAME_OVER` تغییر می‌دهد؛ سپس موتور بازی به سادگی به لایه UI دستور می‌دهد: `renderer.draw_game_over_screen()`.

۴. **استقلال رزولوشن (Responsive Coordinates):**

مختصات رسم اشکال (مثل x و y دکمه‌ها) نباید به صورت اعداد ثابت (Hard-coded) نوشته شوند. همیشه از کسری از `WIDTH` و `HEIGHT` (که از فایل تنظیمات خوانده می‌شوند) استفاده کنید تا با تغییر ابعاد پنجره بازی، گرافیک شما به هم نریزد (مثلاً `x = WIDTH / 2`).
