# راهنمای ماژول موجودیت‌ها و مدل‌های دامنه (Module Entities)
**مسیر در پروژه:** `docs/05_Modules_Documentation/module_entities.md`
**هدف:** تعریف ساختار داده‌های اصلی بازی (مانند بازیکن، سیبل‌ها و آیتم‌ها)، کپسوله‌سازی وضعیت (State) سیستم و ایجاد موجودیت‌هایی که کاملاً از منطق خارجی و گرافیک مستقل هستند.

این سند راهنمای توسعه‌دهنده برای فایل‌های موجود در دایرکتوری `src/entities/` است.

---

## ۱. فلسفه طراحی: مدل‌های دامنه ایزوله (Isolated Domain Models)

در این معماری، «موجودیت‌ها» (Entities) نقش حافظه کوتاه‌مدت سیستم را بازی می‌کنند. آن‌ها نشان‌دهنده اشیاء درون بازی هستند (مثل یک بازیکن که دارای جان و تیر است، یا یک سیبل که دارای مختصات و شعاع است).

مهم‌ترین اصل در طراحی این موجودیت‌ها، **ایزوله‌سازی کامل (Total Isolation)** است. 
موجودیت‌ها هیچ درکی از دنیای بیرون ندارند. کلاس `Player` نمی‌داند که قرار است روی مانیتور رسم شود (بنابراین نیازی به `pygame` ندارد) و نمی‌داند که قرار است در دیتابیس ذخیره شود (بنابراین نیازی به `sqlite3` ندارد). وظیفه یک موجودیت فقط و فقط حفظ داده‌های خودش و محافظت از آن‌ها در برابر تغییرات نامعتبر است.

---

## ۲. ساختار و وظایف زیرماژول‌ها

دایرکتوری `entities` به سه زیربخش اصلی تقسیم می‌شود تا مفاهیم شی‌گرایی (OOP) به درستی پیاده‌سازی شوند:

### 🧬 الف) کلاس‌های پایه (`base/`)
این پوشه شامل کلاس‌های والد (Parent Classes) است که سایر موجودیت‌ها از آن‌ها ارث‌بری می‌کنند.
* **`game_object.py`:** کلاس بنیادین تمام اشیای درون صفحه. هر چیزی که در بازی وجود دارد (نشانگر، سیبل، آیتم) باید دارای مختصات ریاضی (`x` و `y`) و یک وضعیت فعال بودن (`is_active: bool`) باشد. این کلاس کدهای تکراری مختصات‌دهی را از بین می‌برد.

### 👤 ب) مدل‌های بازیکن (`player/`)
محافظت از داده‌های حیاتی کاربر در این بخش انجام می‌شود.
* **`player_model.py`:** این کلاس نگهدارنده وضعیت بازیکن است (امتیاز، زمان باقی‌مانده، تعداد تیر و نام). علاوه بر این، یک متغیر حیاتی به نام `is_crosshair_revealed` (با مقدار اولیه `False`) دارد. بر اساس قوانین بازی، نشانگر در ابتدا نامرئی است و این متغیر تنها پس از شلیک اولین تیر به صورت داخلی به `True` تغییر وضعیت می‌دهد.
* **`crosshair.py`:** مختصات نشانگرِ روی صفحه را در خود نگه می‌دارد و وضعیت‌هایی مثل شعاع تأثیر (Hitbox Radius) را مدیریت می‌کند.

### 🎯 ج) سیبل‌ها و آیتم‌ها (`targets/` و `items/`)
* **`target_base.py`:** موجودیت والد که دارای ویژگی‌هایی مانند `base_score` (امتیاز پایه) و `radius` (شعاع دایره) است.
* **آیتم‌های ویژه (`item_base.py`، `ammo_box.py` و ...):** این کلاس‌ها اطلاعات مختص به آیتم‌های تقویتی یا تخریبی را نگهداری می‌کنند. طبق قوانین معماری این پروژه، **تمام آیتم‌ها باید بلااستثنا فرزند کلاس سیبل باشند** (ارث‌بری مستقیم).

## ۳. کپسوله‌سازی و کنترل وضعیت (Encapsulation)

موجودیت‌ها نباید اجازه دهند هر کلاسی از بیرون مقادیر آن‌ها را مستقیماً تغییر دهد. شما باید از داده‌ها با استفاده از متدهای عمومی (Public Methods) یا ویژگی‌ها (Properties) محافظت کنید. 

**نمونه‌ای از پیاده‌سازی اصولی (کپسوله‌سازی):**
```python
class PlayerModel:
    def __init__(self, username: str, initial_ammo: int, initial_time: float):
        # ویژگی‌ها به صورت خصوصی (Private) تعریف می‌شوند تا مستقیماً دستکاری نشوند
        self._username = username
        self._ammo = initial_ammo
        self._time_remaining = initial_time
        self._is_crosshair_revealed = False

    @property
    def is_locked_out(self) -> bool:
        """بازیکنی که تیر یا زمانش صفر شود، قفل می‌شود."""
        return self._ammo <= 0 or self._time_remaining <= 0.0

    def fire_shot(self) -> bool:
        """
        تلاش برای شلیک. یک تیر کم کرده و نشانگر را مرئی می‌کند.
        خروجی نشان‌دهنده موفقیت یا شکست در شلیک است.
        """
        if self.is_locked_out:
            return False
            
        self._ammo -= 1
        if not self._is_crosshair_revealed:
            self._is_crosshair_revealed = True
            
        return True
````

_در این حالت، هیچ لایه دیگری نمی‌تواند به اشتباه مقدار تیر بازیکن را به `-5` تغییر دهد، زیرا موجودیت از خودش محافظت می‌کند._

## ۴. خطوط قرمز معماری (Strict Constraints)

توسعه‌دهنده لایه موجودیت‌ها موظف است قوانین زیر را با دقت و وسواس کامل رعایت کند:

۱. **ممنوعیت مطلق واردات زیرساختی (Zero Infra Imports):** دقیقاً مانند لایه مکانیک‌ها، فایل‌های این دایرکتوری **تحت هیچ شرایطی** مجاز به `import pygame` یا استفاده از کتابخانه‌های دیتابیس نیستند. اگر موجودیتی نیاز به نمایش رنگ دارد، رنگ باید به صورت یک تاپل ساده مثل `(255, 0, 0)` ذخیره شود، نه به عنوان یک شیء گرافیکی پای‌گیم.

۲. **شکست سریع در سازنده (Fail-fast in **init**):**

موجودیت‌ها نباید با مقادیر نامعتبر ساخته شوند. اگر کسی سعی کرد یک سیبل با شعاع منفی بسازد، کلاس باید بلافاصله سیستم را متوقف کند.

- ❌ **رویکرد غلط:** `self.radius = -10` (و عبور از روی خطا)
- ✅ **رویکرد مهندسی:** `if radius <= 0: raise ValueError("Radius must be positive")`

۳. **پرهیز از موجودیت‌های کم‌خون (Anemic Domain Models):**

موجودیت نباید فقط مجموعه‌ای از `getter` و `setter`های ساده باشد (مثل یک دیکشنری احمقانه). موجودیت باید رفتارهای منطقیِ مربوط به داده‌های خودش را درون خودش داشته باشد (مثل تابع `consume_ammo` در مثال بالا).

۴. **جداسازی کامل از مکانیک‌ها (Separation from Mechanics):**

موجودیت‌ها فقط داده‌ها و آپدیت‌های درونی خود را مدیریت می‌کنند. فرمول‌های ریاضی و منطق بازی **نباید** در کلاس موجودیت نوشته شوند.

- یک کلاس `Target` فقط می‌داند که کجای صفحه قرار دارد (مختصات `x` و `y`).
- کلاس `Target` **نباید** تابعی به نام `check_collision_with_bullet()` داشته باشد. تشخیص برخورد وظیفه لایه `Mechanics` است، نه لایه `Entities`. لایه مکانیک‌ها موجودیت سیبل و موجودیت بازیکن را می‌گیرد و خودش محاسبات تقاطع را انجام می‌دهد.

۵. **الزام به ارث‌بری سلسله‌مراتبی (Strict Inheritance Rule):**
بر اساس نیازمندی‌های قطعی پروژه، ایجاد کلاس‌های مستقل برای آیتم‌ها ممنوع است. تمام آیتم‌های ویژه (مثل زمان اضافه، تیر اضافه یا کسر امتیاز) **باید** به صورت `class Item(Target):` تعریف شوند تا رفتار پایه‌ای سیبل‌ها (مثل ظاهر شدن، ناپدید شدن و داشتن شعاع تقاطع) را به صورت خودکار به ارث ببرند.
