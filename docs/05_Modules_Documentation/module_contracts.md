# راهنمای ماژول قراردادها و اینترفیس‌ها (Module Contracts)
**مسیر در پروژه:** `docs/05_Modules_Documentation/module_contracts.md`
**هدف:** تبیین قوانین «طراحی مبتنی بر قرارداد» (Design by Contract)، ایزوله‌سازی هسته نرم‌افزار از تکنولوژی‌های خارجی و تعریف پورت‌ها (Ports) در معماری هگزاگونال.

این سند راهنمای توسعه فایل‌های موجود در پوشه `src/contracts/` است.

---

## ۱. فلسفه قراردادها (The Contract Philosophy)

در معماری ما، پوشه `contracts` بالاترین سطح انتزاع (Abstraction) را دارد. این پوشه مانند قانون اساسیِ سیستم است. 

هسته بازی (Core) برای انجام کارهای خود به ابزارهای بیرونی نیاز دارد (مثل دیتابیس برای ذخیره امتیاز، یا مانیتور برای رسم تصویر). اما اگر هسته مستقیماً به `SQLite` یا `Pygame` متصل شود، معماری ما در هم می‌شکند. 
راه‌حل این است که هسته، مجموعه‌ای از «قراردادها» یا «اینترفیس‌ها» را تعریف کند و بگوید: *"من با هر کلاسی که این فرم‌ها را امضا کرده باشد و این متدها را داشته باشد، کار می‌کنم؛ برایم مهم نیست درون آن کلاس چه خبر است."*

---

## ۲. ابزار پیاده‌سازی در پایتون (Python Abstraction Tools)

زبان پایتون به صورت بومی کلمه کلیدی `interface` (مانند جاوا یا سی‌شارپ) را ندارد. برای پیاده‌سازی قراردادها، تیم ما منحصراً از کتابخانه داخلی `abc` (Abstract Base Classes) استفاده می‌کند.

**قالب استاندارد یک قرارداد در این پروژه:**
```python
from abc import ABC, abstractmethod
from typing import List
from src.database.dtos import LeaderboardEntryDTO, MatchResultDTO

class IDatabase(ABC):
    """
    قرارداد ارتباط با پایگاه داده.
    هر کلاسی که این قرارداد را امضا می‌کند، موظف است تمام متدهای زیر را پیاده‌سازی نماید.
    """

    @abstractmethod
    def authenticate_or_register(self, username: str, raw_password: str) -> bool:
        """بررسی صحت رمز عبور یا ثبت‌نام کاربر جدید."""
        pass

    @abstractmethod
    def save_match_result(self, match_dto: MatchResultDTO) -> bool:
        """امتیاز بازیکن را در دیتابیس ذخیره می‌کند."""
        pass

    @abstractmethod
    def get_top_scores(self, top_n: int = 10) -> List[LeaderboardEntryDTO]:
        """لیست نفرات برتر را در قالب اشیاء امن DTO برمی‌گرداند."""
        pass
````

## ۳. قراردادهای حیاتی سیستم (Core Interfaces)

توسعه‌دهندگان موظف هستند ۵ قرارداد اصلی زیر را برای ارتباط بین لایه‌ها و رعایت اصل وارونگی وابستگی (DIP) پیاده‌سازی کنند:

### 💾 ۱. قرارداد پایگاه داده (`i_database.py`)

- **نقش:** پورت خروجی برای احراز هویت، ذخیره و بازیابی داده‌ها.
    
- **متدهای الزامی:** * `authenticate_or_register(...)`: برای ورود یکپارچه بازیکنان.
    
    - `save_match_result(...)`: برای ذخیره دیتای پایان بازی.
        
    - `get_top_scores(...)`: برای استخراج دیتای ایمن (DTO) جهت صفحه Leaderboard.
        

### 🎨 ۲. قرارداد رندرینگ (`i_renderable.py`)

- **نقش:** پورت خروجی برای رسم گرافیک روی مانیتور (ایزوله کردن از Pygame).
    
- **متدهای الزامی:**
    
    - `clear_screen(color: tuple)`: پاک کردن فریم قبلی.
        
    - `draw_entity(x: float, y: float, radius: float, color: tuple)`: رسم سیبل‌ها و بازیکن.
        
    - `draw_ui_text(text: str, x: float, y: float)`: رسم متون HUD.
        
    - `flip_buffer()`: نمایش فریم آماده‌شده به کاربر.
        

### 🔊 ۳. قرارداد پخش‌کننده صوتی (`i_audio_player.py`)

- **نقش:** پورت خروجی برای پخش افکت‌های صوتی (SFX) بدون درگیر کردن هسته با کتابخانه‌های صوتی.
    
- **متدهای الزامی:**
    
    - `play_sfx(sound_name: str)`: دریافت نام یک افکت صوتی (مثل شلیک یا برخورد) و پخش آن.
        
    - `play_background_music(track_name: str)`: پخش موسیقی در یک حلقه بی‌نهایت.
        
    - `stop_all_sounds()`: قطع تمام صداها (مناسب برای زمان پایان بازی).
        

### 🎯 ۴. قرارداد تقاطع و فیزیک (`i_collidable.py`)

- **نقش:** رابط استاندارد برای هر موجودیتی (مثل سیبل‌ها یا آیتم‌ها) که می‌تواند مورد اصابت تیر قرار گیرد.
    
- **متدهای الزامی:**
    
    - `get_hitbox() -> tuple`: برگرداندن مختصات و شعاع حساس به برخورد.
        
    - `is_hit(x: float, y: float) -> bool`: دریافت مختصات شلیک و بررسی ریاضی تقاطع آن با شئ.
        

### 🔄 ۵. قرارداد وضعیت‌ها (`i_state.py`)

- **نقش:** پیاده‌سازی الگوی طراحی State برای ایزوله کردن منطق فازهای مختلف بازی (مثل منو، احراز هویت، جریان بازی).
    
- **متدهای الزامی:**
    
    - `enter()`: کدهایی که فقط یک‌بار هنگام ورود به این وضعیت اجرا می‌شوند.
        
    - `update(dt: float, commands: list)`: اجرای منطق ریاضی و دریافت فرمان‌ها در هر فریم.
        
    - `render(renderer: IRenderable)`: ارسال دستورات رسم گرافیک مربوط به این وضعیت به لایه UI.
        
    - `exit()`: پاک‌سازی متغیرها پیش از خروج از این وضعیت.
        

## ۴. خطوط قرمز معماری در قراردادها (Strict Constraints)

پوشه `contracts` حساس‌ترین بخش معماری است. کوچکترین اشتباه در اینجا، کل سیستم را آلوده می‌کند. توسعه‌دهنده این فایل‌ها باید به شدت مراقب قوانین زیر باشد:

۱. **ممنوعیت مطلق پیاده‌سازی (No Implementation Logic):**

هیچ تابعی در این فایل‌ها حق ندارد حتی یک خط کد اجرایی (لاجیک، فرمول ریاضی یا پرینت) داشته باشد. بدنه تمام متدها باید فقط و فقط شامل کلمه کلیدی `pass` یا `...` (Ellipsis) باشد.

۲. **ایزوله بودن واردات (No Infra Imports):**

فایل‌های این پوشه حق ندارند هیچ کتابخانه خارجی یا زیرساختی را `import` کنند.

- ❌ **شدیداً غلط:** `import pygame` یا `import sqlite3` درون اینترفیس.
    
- ✅ **درست:** استفاده از تایپ‌های بومی پایتون (`str`, `int`, `List`) یا ایمپورت کردن اشیاء انتقال داده از لایه دیتابیس (`DTOs`).
    

۳. **مستندسازی به جای کدنویسی (Docstrings are Mandatory):**

چون در اینترفیس‌ها هیچ کدی وجود ندارد که برنامه‌نویسِ لایه آداپتور بتواند با خواندن آن متوجه عملکرد متد بشود، **نوشتن Docstring برای تک‌تک متدها اجباری است**. Docstring باید دقیقاً توضیح دهد که پیش‌شرط‌ها و پس‌شرط‌های این تابع چیست (Design by Contract).

۴. **تایپ هینتینگ اجباری (Strict Type Hinting):**

یک قرارداد بدون مشخص کردن تایپ ورودی‌ها و خروجی‌ها بی‌ارزش است. تمام متدها باید نوع دقیق آرگومان‌های ورودی و خروجی (مانند `-> bool` یا `-> List`) را مشخص کنند تا IDE (مانند ابزار Mypy) بتواند در صورت تخلفِ کلاسِ پیاده‌ساز، خطا بدهد.
