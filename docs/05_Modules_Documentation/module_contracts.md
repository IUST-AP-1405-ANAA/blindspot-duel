# راهنمای ماژول قراردادها و اینترفیس‌ها (Module Contracts)
**مسیر در پروژه:** `docs/04_Modules_Documentation/module_contracts.md`
**هدف:** تبیین قوانین «طراحی مبتنی بر قرارداد» (Design by Contract)، ایزوله‌سازی هسته نرم‌افزار از تکنولوژی‌های خارجی و تعریف پورت‌ها (Ports) در معماری هگزاگونال.

این سند راهنمای توسعه فایل‌های موجود در پوشه `src/contracts/` است.

---

## ۱. فلسفه قراردادها (The Contract Philosophy)

در معماری ما، پوشه `contracts` بالاترین سطح انتزاع (Abstraction) را دارد. این پوشه مانند قانون اساسیِ سیستم است. 

هسته بازی (Core) برای انجام کارهای خود به ابزارهای بیرونی نیاز دارد (مثل دیتابیس برای ذخیره امتیاز، یا مانیتور برای رسم تصویر). اما اگر هسته مستقیماً به `SQLite` یا `Pygame` متصل شود، معماری ما در هم می‌شکند. 
راه‌حل این است که هسته، مجموعه‌ای از «قراردادها» یا «اینترفیس‌ها» را تعریف کند و بگوید: *"من با هر کلاسی که این فرم‌ها را امضا کرده باشد و این متدها را داشته باشد، کار می‌کنم؛ برایم مهم نیست درون آن کلاس چه خبر است."*

---

## ۲. ابزار پیاده‌سازی در پایتون (Python Abstraction Tools)

زبان پایتون به صورت بومی کلمه کلیدی `interface` (مانند جاوا یا سی‌شارپ) را ندارد. برای پیاده‌سازی قراردادها، تیم ما منحصراً از کتابخانه داخلی `abc` (Abstract Base Classes) استفاده می‌کند.

**قالب استاندارد یک قرارداد در این پروژه:**
```python
from abc import ABC, abstractmethod
from typing import List, Dict

class IDatabase(ABC):
    """
    قرارداد ارتباط با پایگاه داده.
    هر کلاسی (مثلاً SQLiteRepository) که این قرارداد را امضا می‌کند،
    موظف است تمام متدهای زیر را پیاده‌سازی نماید.
    """

    @abstractmethod
    def save_player_score(self, player_name: str, score: int, combo: int) -> bool:
        """
        امتیاز بازیکن را در دیتابیس ذخیره می‌کند.
        خروجی True به معنای موفقیت‌آمیز بودن عملیات است.
        """
        pass

    @abstractmethod
    def get_leaderboard(self, top_n: int = 10) -> List[Dict]:
        """
        لیست نفرات برتر را برمی‌گرداند.
        """
        pass
````

## ۳. قراردادهای حیاتی سیستم (Core Interfaces)

مالک این لایه موظف است حداقل ۳ قرارداد اصلی زیر را برای ارتباط بین لایه‌ها طراحی و تعریف کند:

### 💾 ۱. قرارداد پایگاه داده (`i_database.py`)

- **نقش:** پورت خروجی برای ذخیره و بازیابی داده‌ها.
    
- **متدهای الزامی:** * `save_match_result(...)`: برای ذخیره دیتای پایان بازی.
    
    - `get_top_scores(...)`: برای استفاده در صفحه Leaderboard.
        
- **آداپتور مجری:** کلاس `SQLiteRepository` در لایه دیتابیس (متعلق به `aa-jafari`).
    

### 🎨 ۲. قرارداد رندرینگ (`i_renderable.py`)

- **نقش:** پورت خروجی برای رسم گرافیک روی مانیتور.
    
- **متدهای الزامی:**
    
    - `clear_screen(color: tuple)`: پاک کردن فریم قبلی.
        
    - `draw_entity(x: float, y: float, entity_type: str)`: رسم سیبل‌ها و بازیکن.
        
    - `draw_ui_text(text: str, x: float, y: float)`: رسم متون HUD.
        
    - `flip_buffer()`: نمایش فریم آماده‌شده به کاربر.
        
- **آداپتور مجری:** کلاس‌های لایه `ui` (متعلق به `AmiraliAfshari`).
    

### ⌨️ ۳. قرارداد کنترلر رویدادها (`i_event_handler.py`)

- **نقش:** پورت ورودی برای دریافت دکمه‌های فشرده شده.
    
- **متدهای الزامی:**
    
    - `poll_events() -> List[str]`: خواندن رویدادهای کیبورد/موس و برگرداندن یک لیست از فرمان‌های استاندارد (مثل `["SHOOT", "MOVE_LEFT"]`).
        
- **آداپتور مجری:** کلاسی در لایه Core یا UI که وظیفه تبدیل `pygame.event` به متن‌های استاندارد را دارد.
    

## ۴. خطوط قرمز معماری در قراردادها (Strict Constraints)

پوشه `contracts` حساس‌ترین بخش معماری است. کوچکترین اشتباه در اینجا، کل سیستم را آلوده می‌کند. توسعه‌دهنده این فایل‌ها باید به شدت مراقب قوانین زیر باشد:

۱. **ممنوعیت مطلق پیاده‌سازی (No Implementation Logic):**

هیچ تابعی در این فایل‌ها حق ندارد حتی یک خط کد اجرایی (لاجیک، فرمول ریاضی یا پرینت) داشته باشد. بدنه تمام متدها باید فقط و فقط شامل کلمه کلیدی `pass` یا `...` (Ellipsis) باشد.

۲. **ایزوله بودن واردات (No Infra Imports):**

فایل‌های این پوشه حق ندارند هیچ کتابخانه خارجی یا زیرساختی را `import` کنند.

- ❌ **شدیداً غلط:** `import pygame` یا `import sqlite3` درون اینترفیس.
    
- ✅ **درست:** استفاده از تایپ‌های بومی پایتون (`str`, `int`, `List`) یا ایمپورت کردن اشیاء انتقال داده (DTOs).
    

۳. **مستندسازی به جای کدنویسی (Docstrings are Mandatory):**

چون در اینترفیس‌ها هیچ کدی وجود ندارد که برنامه نویسِ لایه آداپتور بتواند با خواندن آن متوجه عملکرد متد بشود، **نوشتن Docstring برای تک‌تک متدها اجباری است**. Docstring باید دقیقاً توضیح دهد که پیش‌شرط‌ها و پس‌شرط‌های این تابع چیست (Design by Contract).

۴. **تایپ هینتینگ اجباری (Strict Type Hinting):**

یک قرارداد بدون مشخص کردن تایپ ورودی‌ها و خروجی‌ها بی‌ارزش است. تمام متدها باید نوع دقیق آرگومان‌های ورودی و `-> return_type` را مشخص کنند تا IDE بتواند در صورت تخلفِ کلاسِ پیاده‌ساز، خطا بدهد.
