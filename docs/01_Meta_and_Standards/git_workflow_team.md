# استراتژی مدیریت نسخه و همکاری تیمی (Git Workflow)
**حوزه کاربرد:** مدیریت سورس‌کد، جلوگیری از تداخل کدهای اعضای تیم (Merge Conflicts) و استانداردسازی تاریخچه توسعه.
**مخاطب سند:** تمام اعضای تیم (از برنامه‌نویسان تازه‌کار تا معماران نرم‌افزار).

---

## مقدمه: چرا به یک پروتکل گیت نیاز داریم؟
گیت (Git) صرفاً یک سیستم پشتیبان‌گیری (Backup) یا فضای ابری برای ذخیره کدها نیست؛ گیت یک **پروتکل همکاری ناهمگام (Asynchronous Collaboration)** است. در یک تیم ۴ نفره، اگر همه اعضا همزمان کدهای خود را روی یک فایل ذخیره کنند، فاجعه‌ای از تداخل‌ها رخ می‌دهد. 

این سند رویکرد **GitHub Flow** را به عنوان استاندارد تعریف می‌کند. این رویکرد به شما اجازه می‌دهد در یک محیط کاملاً ایزوله و امن کدهای خود را توسعه دهید و تنها زمانی که از صحت عملکرد آن‌ها مطمئن شدید، آن‌ها را به هسته اصلی پروژه تزریق کنید.

---

## ۱. معماری شاخه‌ها (Branching Strategy)

**مفهوم پایه برای تازه‌کارها:** یک «شاخه» (Branch) مانند یک خط زمانی موازی است. وقتی شما یک شاخه جدید می‌سازید، در واقع یک کپی دقیق از آخرین وضعیت پروژه می‌گیرید تا تغییرات خود را در آن اعمال کنید، بدون اینکه کدهای اصلی (که بقیه در حال استفاده از آن هستند) خراب شود.

### قوانین شاخه‌ها:
* **شاخه `main` (خط قرمز تیم):** این شاخه همیشه و در هر لحظه باید حاوی کدهای پایدار، بدون باگ و قابل اجرا باشد. **کامیت مستقیم (Direct Push) به شاخه `main` مطلقاً ممنوع است.**
* **شاخه‌های توسعه (Feature Branches):** هر عضو تیم برای انجام هر وظیفه (تسک)، باید یک شاخه جدید از روی `main` بسازد و کار خود را منحصراً در آن شاخه انجام دهد.

### استاندارد نام‌گذاری شاخه‌ها:
نام شاخه باید دقیقاً مشخص کند که چه نوع تغییری در حال انجام است.
* `feat/name-of-feature` -> برای اضافه کردن یک قابلیت جدید (مثال: `feat/add-scoring-system`)
* `fix/name-of-bug` -> برای رفع یک باگ (مثال: `fix/crosshair-invisible-bug`)
* `docs/name-of-doc` -> برای تغییرات مستندات (مثال: `docs/update-readme`)
* `refactor/name-of-module` -> برای تمیز کردن و بازنویسی کد بدون تغییر در رفتار آن.

## ۲. استاندارد پیام‌های کامیت (Conventional Commits)

**مفهوم پایه برای تازه‌کارها:** «کامیت» (Commit) مانند زدن دکمه Save در یک بازی است که یک نقطه بازگشت (Checkpoint) ایجاد می‌کند. پیام کامیت باید به بقیه تیم بگوید دقیقاً چه تغییری در این Checkpoint رخ داده است.

ما از استاندارد *Conventional Commits* پیروی می‌کنیم. پیام‌های مبهم مثل `update file` یا `fixed stuff` اکیداً ممنوع هستند.

**ساختار استاندارد:**
`نوع_تغییر(بخش_مربوطه): توضیحات دقیق و کوتاه`

**مثال‌های صحیح (✅ رویکرد مهندسی):**
* `feat(mechanics): add combo point multiplier for consecutive hits`
* `fix(ui): correct overlapping text in leaderboard screen`
* `docs(architecture): add component diagram to docs vault`

**مثال‌های غلط (❌ رویکرد ضعیف):**
* `added points` (نامشخص بودن نوع تغییر و محل آن)
* `fixed a bug with the crosshair` (عدم استفاده از پیشوند استاندارد)

## ۳. چرخه کار روزانه (Daily Workflow - گام‌به‌گام)

این بخش دستورالعمل اجرایی شما برای شروع و پایان کار در هر روز است. فرض کنید وظیفه شما ایجاد «سیستم امتیازدهی» است.

### گام صفر: انتخاب وظیفه و به‌روزرسانی کانبان (Kanban Board)
پیش از نوشتن حتی یک خط کد، باید شفافیت ایجاد کنید:
۱. به بورد کانبان پروژه در تب `Projects` گیت‌هاب بروید.
۲. یکی از کارت‌های اختصاص یافته به خودتان را از ستون `Todo` برداشته و به ستون `In Progress` بکشید (درَگ کنید). این کار به بقیه تیم نشان می‌دهد: *"من در حال حاضر مشغول کدنویسی این فایل هستم، لطفا سمت آن نروید!"*

### گام اول: همگام‌سازی با مخزن اصلی (Sync)
همیشه روز کاری خود را با دریافت آخرین کدهای تایید شده از هم‌تیمی‌ها آغاز کنید.
```bash
git checkout main
git pull origin main
````

### گام دوم: ایجاد شاخه ایزوله (Branch Creation)

هرگز روی `main` کد ننویسید. یک شاخه جدید بسازید و وارد آن شوید:
```bash
git checkout -b feat/scoring-system
```

### گام سوم: توسعه و ثبت تغییرات (Stage & Commit)

کدهای خود را بنویسید. وقتی یک بخش منطقی کامل شد (مثلاً فرمول ریاضی را نوشتید)، تغییرات را ثبت کنید. کامیت‌ها باید اتمیک (کوچک و متمرکز) باشند.

```bash
# اضافه کردن فایل‌های تغییر یافته به استیج
git add src/mechanics/scoring/math_formulas.py

# ثبت تغییرات با پیام استاندارد
git commit -m "feat(scoring): implement euclidean distance math formula"
```

### گام چهارم: همگام‌سازی مجدد و ارسال به سرور (Pull & Push)

پیش از ارسال کدها، **بسیار حیاتی** است که آخرین تغییرات هم‌تیمی‌هایتان را از `main` دریافت کنید تا مطمئن شوید کدهایتان با کدهای دیروزِ بقیه تضاد (Conflict) ندارد:

```bash
# ۱. دریافت آخرین تغییرات شاخه اصلی در شاخه فعلی خودتان
git pull origin main

# ۲. در صورت عدم وجود تداخل، کدهایتان را به گیت‌هاب بفرستید:
git push origin feat/scoring-system
```

### گام پنجم: درخواست ادغام (Pull Request / PR)

به ریپازیتوری پروژه در GitHub بروید. یک `Pull Request` باز کنید.

یک **PR چیست؟** یک درخواست رسمی از هم‌تیمی‌هاست که می‌گوید: _"من وظیفه‌ام را در شاخه ایزوله‌ام تمام کرده‌ام. لطفاً کدهای مرا بررسی کنید تا در صورت تایید، به شاخه `main` اضافه (Merge) شود."_

**نکته بسیار مهم (بستن خودکار ایشوها):** در توضیحات PR خود حتماً از کلمات کلیدی گیت‌هاب همراه با شماره ایشو استفاده کنید (مثلاً بنویسید `Closes #12` یا `Fixes #12`). با این کار، به محض اینکه کدهای شما تایید و Merge شود، گیت‌هاب به طور خودکار آن ایشو را می‌بندد و کارت آن را در بورد کانبان به ستون `Done` منتقل می‌کند.

**قوانین ادغام (Merge Rules):**
۱. **فقط Squash and merge:** هنگام تایید PR در گیت‌هاب، حتماً روی فلش کنار دکمه Merge کلیک کرده و گزینه `Squash and merge` را انتخاب کنید. این کار باعث می‌شود تمام کامیت‌های ریزِ آزمایشیِ شما تبدیل به یک کامیت یکپارچه و تمیز در شاخه `main` شود.
۲. **حذف شاخه (Housekeeping):** بلافاصله پس از Merge شدن، روی دکمه `Delete branch` کلیک کنید تا ریپازیتوری پر از شاخه‌های اضافی و متروکه نشود.

## ۴. قوانین بازبینی کد (Code Review) و ادغام

برای تضمین کیفیت نرم‌افزار (Quality Assurance) و رعایت پروتکل‌های معماری، قوانین زیر در GitHub اعمال می‌شوند:

۱. **قانون دو جفت چشم (Four Eyes Principle):** هیچکس اجازه ندارد Pull Request خودش را تایید (Approve) و Merge کند.

۲. **سیستم چرخشی (Round-Robin Review):** برای جلوگیری از گلوگاه شدن یک نفر، شما موظف هستید هنگام ساخت PR، دقیقاً شخص تعیین‌شده را به عنوان `Reviewer` در گیت‌هاب انتخاب کنید:
   * کدهای `arsalandalvand` -> بازبین: `Navidkhd`
   * کدهای `Navidkhd` -> بازبین: `AmiraliAfshari`
   * کدهای `AmiraliAfshari` -> بازبین: `aa-jafari`
   * کدهای `aa-jafari` -> بازبین: `arsalandalvand`

۳. **بررسی معماری:** بازبین (Reviewer) موظف است چک کند که کدهای جدید ناقض قوانین پوشه `Development_Protocols/` (مثل اصل DRY یا Law of Demeter) و دیگر قوانین اعلام شده نباشند. در صورت وجود مشکل، کامنت می‌گذارد و درخواست تغییر (Request Changes) می‌دهد.

## ۵. چگونه با تداخل کدها (Merge Conflicts) برخورد کنیم؟

تداخل زمانی رخ می‌دهد که دو نفر همزمان روی یک خط از یک فایل تغییراتی ایجاد کنند. برای جلوگیری از این کابوس:

- **تعهد به WBS:** دقیقاً روی فایل‌هایی کار کنید که در سند `work_breakdown_structure.md` به شما محول شده است. اگر وظیفه شما بخش دیتابیس است، حق ندارید فایل‌های مربوط به UI را دستکاری کنید.
- **کامیت‌های کوچک و مکرر:** کدهای خود را یک هفته روی سیستم خود نگه ندارید. هر روز کارهای تکمیل‌شده را Push کنید و PR بسازید.
- **ارتباط تیمی:** اگر مجبور شدید فایلی خارج از دامنه وظایف خود را تغییر دهید (مثلاً یک تابع عمومی در لایه `utils`)، حتماً پیش از تغییر در گروه تیمی اطلاع‌رسانی کنید.

**اگر تداخل رخ داد چه کار کنیم؟**
نترسید! هرگز فایل‌هایی که گیت در آن‌ها علائم `<<<<<<< HEAD` تولید کرده را مستقیماً کامیت نکنید. فایل را در ادیتور VSCode باز کنید، سیستم به طور خودکار بخش‌های تداخل را رنگی نشان می‌دهد. یکی از گزینه‌های `Accept Current Change` (کد خودتان) یا `Accept Incoming Change` (کد هم‌تیمی) را با دقت انتخاب کنید و سپس فایل را ذخیره (`Save`) کرده و تغییرات را مجدداً کامیت کنید.
