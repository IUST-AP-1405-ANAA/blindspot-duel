# مدیریت وضعیت و تغییرات (State Management and Mutation)
**حوزه کاربرد:** این پروتکل برای هر سیستمی که داده‌های آن در طول زمان تغییر می‌کنند (مانند بازی‌ها، رابط‌های کاربری تعاملی و سیستم‌های پردازش همزمان) الزامی است.

---

## ۱. ریشه پیچیدگی: وضعیت‌های کنترل‌نشده (Uncontrolled State)

**مفهوم معماری:** «وضعیت» یا State به تمام داده‌هایی گفته می‌شود که نرم‌افزار در یک لحظه خاص در حافظه نگه می‌دارد (مثل امتیاز فعلی بازیکن، وضعیت لاگین، یا دیتای کش‌شده). باگ‌های ردیابی وضعیت (State Bugs) زمانی رخ می‌دهند که بخش‌های مختلف برنامه، به صورت همزمان و بدون نظارت، این داده‌ها را تغییر دهند (Mutation).

**چرا اهمیت دارد؟** تغییر دادن مستقیم متغیرها (به ویژه متغیرهای عمومی یا اشیای به اشتراک گذاشته شده) باعث رفتارهای غیرقابل پیش‌بینی در سیستم می‌شود. دیباگ کردن سیستمی که مقادیر آن دائماً توسط توابع ناشناس دستکاری می‌شود، در پروژه‌های بزرگ به یک کابوس مهندسی تبدیل می‌گردد.

---

## ۲. تغییرناپذیری (Immutability)

**مفهوم معماری:** تغییرناپذیری به این معناست که وقتی یک شیء یا ساختار داده ایجاد شد، دیگر **هرگز** نباید مقادیر داخلی آن تغییر کند. برای اعمال یک تغییر، باید یک «کپی کاملاً جدید» از آن شیء با مقادیر به‌روزرسانی‌شده ایجاد شود.

**چرا اهمیت دارد؟** آبجکت‌های تغییرناپذیر (Immutable Objects) ذاتاً برای پردازش‌های همزمان (Thread-safe) ایمن هستند. از آنجا که هیچ تابعی نمی‌تواند دیتای آن‌ها را مخفیانه تغییر دهد، نیازی به قفل کردن حافظه (Mutex/Locks) نیست و جریان داده در سیستم کاملاً شفاف و قابل پیش‌بینی می‌شود.

**الزام اجرایی و مثال:**
* ❌ **رویکرد ضعیف (Mutation):** تغییر مستقیم فیلدهای یک آبجکت.
```python
  def update_player_score(player: Player, points: int):
      # فاجعه: وضعیت داخلی شیء به صورت مستقیم و درجا تغییر کرد
      player.score += points 
      player.last_updated = time.now()
```

- ✅ **رویکرد مهندسی (Immutability):** برگرداندن یک شیء جدید (استفاده از `dataclasses` یا رویکردهای مشابه).
```Python
    def calculate_new_score(player: Player, points: int) -> Player:
        # ایجاد یک کپی جدید از بازیکن با امتیاز جدید؛ شیء قبلی دست‌نخورده می‌ماند
        return replace(player, score=player.score + points, last_updated=time.now())
```

## ۳. جداسازی فرمان و پرسش (CQS - Command-Query Separation)

**مفهوم معماری:** این اصل بیان می‌کند که هر متد در سیستم باید فقط یکی از این دو حالت باشد:

۱. **فرمان (Command):** وضعیت سیستم را تغییر می‌دهد (عملیات انجام می‌دهد) اما **هیچ مقداری برنمی‌گرداند** (خروجی آن Void/None است).

۲. **پرسش (Query):** اطلاعاتی را محاسبه کرده و برمی‌گرداند، اما **وضعیت سیستم را به هیچ‌وجه تغییر نمی‌دهد**.

**چرا اهمیت دارد؟** پرسش‌ها (Queries) نباید دارای عوارض جانبی (Side-effects) باشند. اگر شما صد بار از دیتابیس بپرسید "موجودی این کاربر چقدر است؟"، این پرسش نباید باعث کسر شدن کارمزد از حساب او شود! ترکیب کردن فرمان و پرسش باعث ایجاد توابعی می‌شود که رفتار پنهان و مخرب دارند.

**الزام اجرایی و مثال:**

- ❌ **رویکرد ضعیف (ترکیب Command و Query):**
```Python
    def get_and_decrement_ammo(self) -> int:
        self.ammo -= 1           # Command: در حال تغییر وضعیت
        return self.ammo         # Query: در حال برگرداندن مقدار
```

- ✅ **رویکرد مهندسی (جداسازی دقیق):**
```Python
    def decrement_ammo(self) -> None:
        self.ammo -= 1           # Command: فقط تغییر وضعیت (بدون خروجی)
    
    def get_current_ammo(self) -> int:
        return self.ammo         # Query: فقط خواندن وضعیت (بدون تغییر)
```

## ۴. توابع خالص (Pure Functions) و ایزوله‌سازی عوارض جانبی

**مفهوم معماری:** یک تابعِ خالص (Pure Function) تابعی است که:

۱. خروجی آن _صرفاً_ به ورودی‌هایش بستگی دارد (اگر ورودی یکسان بدهید، همیشه خروجی یکسان می‌گیرید).

۲. هیچ‌گونه تعاملی با دنیای بیرون ندارد و وضعیت خارجی را تغییر نمی‌دهد (بدون Side-effect). یعنی نه در فایل می‌نویسد، نه به شبکه متصل می‌شود و نه متغیرهای Global را تغییر می‌دهد.

**چرا اهمیت دارد؟** توابع خالص بی‌نهایت پایدار هستند و تست نوشتن (Unit Testing) برای آن‌ها در چند ثانیه انجام می‌شود، زیرا نیازی به ساختن دیتابیسِ فیک (Mocking) ندارند.

**الزام اجرایی و مثال:**

تیم موظف است تا حد امکان منطق ریاضی و بیزینسی را به شکل توابع خالص بنویسد و عوارض جانبی (مثل ذخیره در دیتابیس یا رسم روی صفحه) را به مرزهای بیرونی سیستم (لایه UI یا Database) هل دهد.

- ❌ **رویکرد ضعیف (تابع ناخالص - Impure):**

```Python
    def calculate_and_save_damage(damage: int):
        final_damage = damage * 1.5
        database.save(final_damage) # Side-effect: اتصال به دیتابیس درون تابع ریاضی!
        return final_damage
```

- ✅ **رویکرد مهندسی (تابع خالص - Pure):**
```Python
    def calculate_damage_multiplier(damage: int) -> float:
        return damage * 1.5 # فقط محاسبه ریاضی؛ ۱۰۰٪ قابل تست و مستقل.
    
    # ذخیره در دیتابیس در یک لایه دیگر (مثلاً در GameEngine) انجام می‌شود.
```

## ۵. منبع حقیقت واحد (SSOT - Single Source of Truth)

**مفهوم معماری:** هر قطعه از دیتای وضعیت (مثل "نام بازیکن" یا "امتیاز فعلی") باید فقط و فقط در یک مکان مشخص از حافظه ذخیره و مدیریت شود.

**چرا اهمیت دارد؟** اگر امتیاز بازیکن را هم در کلاس `Player` و هم به عنوان یک متغیر جداگانه در کلاس `UI_HUD` ذخیره کنید، در نهایت این دو متغیر از سینک (Sync) خارج می‌شوند و رابط کاربری امتیازی متفاوت از واقعیت را نشان خواهد داد.

**الزام اجرایی:**

کلاس‌های رابط کاربری (UI) حق ندارند وضعیت‌ها را درون خود ذخیره (Cache) کنند. آن‌ها باید مقادیر را به صورت لحظه‌ای از کلاس‌های هسته (مدل‌ها) استعلام بگیرند یا از طریق الگوی Observer (مثل رویدادها/Events) به تغییرات گوش دهند.
