# تاب‌آوری و کدنویسی تدافعی (Resilience and Defensive Code)
**مسیر در پروژه:** `Development_Protocols/06_Resilience_and_Defensive_Code.md`
**حوزه کاربرد:** این پروتکل تضمین می‌کند که نرم‌افزار در مواجهه با داده‌های نامعتبر، خطاهای شبکه‌ای یا اشتباهات توسعه‌دهندگان، رفتاری پایدار و قابل پیش‌بینی داشته باشد و از فروپاشی‌های زنجیره‌ای (Cascading Failures) جلوگیری کند.

---

## ۱. اصل شکست سریع (Fail-Fast Principle)

**مفهوم معماری:** نرم‌افزار باید در لحظه‌ای که متوجه یک وضعیت نامعتبر، کمبود منابع یا دیتای غیرمنطقی شد، عملیات را بلافاصله متوقف کرده و خطا را گزارش دهد. تلاش برای «ادامه دادن به هر قیمتی» با داده‌های ناقص، به مراتب مخرب‌تر از کرش کردن (Crash) برنامه است. درواقع برنامه نویس در هرجایی از برنامه باید کد را بصورتی که درصورتی برنامه با ارور مواجه شد یا اتفاقی غیرمنتظره رخ داد برنامه سریعا لاگ و گزارشش را ارسال کند و بر مبنای دیتای خراب به کار خود ادامه ندهد.

**چرا اهمیت دارد؟** سیستمی که خطا را نادیده می‌گیرد و به کار خود ادامه می‌دهد، وارد «وضعیت زامبی» (Zombie State) می‌شود. در این حالت، ممکن است دیتابیس با داده‌های فاسد (Corrupted Data) پر شود و ریشه‌یابیِ (Debugging) خطایی که ده‌ها مرحله قبل رخ داده، عملاً غیرممکن گردد.

**الزام اجرایی و مثال:**
اگر برنامه‌ای برای اجرای صحیح به یک فایل تنظیمات (Config) نیاز دارد، و فایل وجود ندارد، برنامه نباید با مقادیر پیش‌فرضِ حدس‌زده‌شده بالا بیاید؛ بلکه باید در همان ثانیه اول (Initialization) اجرا را متوقف کند.
* ❌ **رویکرد ضعیف (تلاش برای بقا با دیتای ناقص):**
```python
  def process_payment(amount: float):
      if amount < 0:
          print("Warning: Amount is negative. Setting to zero.")
          amount = 0  # فاجعه: سرپوش گذاشتن روی یک باگ منطقی در جای دیگر برنامه!
      execute_transaction(amount)
```

- ✅ **رویکرد مهندسی (شکست سریع):**
```Python
    def process_payment(amount: float):
        if amount <= 0:
            # بلافاصله متوقف می‌شود تا برنامه‌نویس متوجه شود کجای سیستم مقدار منفی تولید کرده است.
            raise ValueError(f"Invalid payment amount: {amount}") 
        execute_transaction(amount)
```

## ۲. الگوی خروج زودهنگام (Early Return / Bouncer Pattern)

**مفهوم معماری:** توابع باید در همان خطوط ابتدایی، تمام پیش‌شرط‌ها (Pre-conditions) و موارد نامعتبر را بررسی کرده و از تابع خارج شوند (Return یا Throw Exception). مسیر اصلی و موفقیت‌آمیز تابع (Happy Path) نباید داخل بلاک‌های شرطی (if) قرار بگیرد.

**چرا اهمیت دارد؟** استفاده از `if/else`های تودرتو باعث ایجاد کد ضدالگوی پیکان (Arrow Anti-Pattern) می‌شود که خوانایی کد را نابود کرده و پیگیری مسیر اجرای برنامه را برای ذهن انسان دشوار می‌کند.

**الزام اجرایی و مثال:**

- ❌ **رویکرد ضعیف (کد تودرتو و ناخوانا):**

```Python
    def apply_discount(user, discount_code):
        if user is not None:
            if user.is_active:
                if discount_code.is_valid():
                    # مسیر اصلی برنامه (Happy Path) در عمق فرورفتگی گیر افتاده است
                    user.balance -= discount_code.value
                    return True
        return False
```

- ✅ **رویکرد مهندسی (خروج زودهنگام - Bouncer Pattern):**
```Python
    def apply_discount(user, discount_code):
        # مانند یک نگهبان دم در (Bouncer)، موارد نامعتبر را فیلتر می‌کنیم
        if user is None or not user.is_active:
            return False
        if not discount_code.is_valid():
            return False
    
        # مسیر اصلی برنامه در بالاترین سطح (بدون فرورفتگی) و با وضوح کامل اجرا می‌شود
        user.balance -= discount_code.value
        return True
```

## ۳. ایمنی در برابر مقادیر پوچ (Null Safety / The Billion Dollar Mistake)

**مفهوم معماری:** برگرداندن مقدار `Null` یا `None` از توابع، یکی از بزرگترین دلایل فروپاشی نرم‌افزارها (NullPointerException) در محیط پروداکشن (Production) است.

**چرا اهمیت دارد؟** وقتی تابعی `Null` برمی‌گرداند، تمام ماژول‌هایی که از آن تابع استفاده می‌کنند مجبورند دائماً چک کنند که `if result is not None`. فراموش کردن حتی یک شرط، باعث کرش کردن کل سیستم می‌شود.

**الزام اجرایی و مثال:**

- جایگزین کردن `Null` با الگوهای ایمن: توابعی که یک لیست/مجموعه برمی‌گردانند، در صورت نبود داده، باید یک لیست خالی (Empty Collection) برگردانند، نه `Null`.

- ❌ **رویکرد ضعیف:**
```Python
    def get_active_users():
        users = database.fetch_users("active")
        if not users:
            return None # خطرناک: کلاینت در صورت انجام حلقه روی این مقدار، کرش می‌کند
        return users
```

- ✅ **رویکرد مهندسی:**
```Python
    def get_active_users() -> list:
        users = database.fetch_users("active")
        if not users:
            return [] # ایمن: کلاینت حلقه را اجرا می‌کند اما هیچ اتفاقی نمی‌افتد (بدون کرش)
        return users
```

## ۴. مدیریت خطای اصولی و ممنوعیت بلعیدن خطا (No Error Swallowing)

**مفهوم معماری:** استثناءها (Exceptions) ابزارهای ارتباطی سیستم هستند که می‌گویند "من نتوانستم به تعهدم عمل کنم". گرفتن (Catch) یک خطا و بی‌تفاوت رد شدن از آن، بدترین نوع کدنویسی است.

**چرا اهمیت دارد؟** بلعیدن خطا باعث می‌شود نرم‌افزار به ظاهر درست کار کند، در حالی که پردازش‌ها در پس‌زمینه در حال شکست خوردن هستند. این کار، مانیتورینگ سیستم و دیباگ را مختل می‌کند.

**الزامات اجرایی و مثال:**
۱. هرگز از بلوک‌های خالی (مانند `pass` در پایتون) برای مدیریت استثناء استفاده نکنید. اگر خطایی Catch می‌شود، باید لاگ (Log) شود یا به صورت اصولی مدیریت گردد.
۲. همیشه خطاهای تخصصی (Specific Exceptions) را شکار کنید، نه کلاس عمومی `Exception` را.

- ❌ **رویکرد ضعیف (بلعیدن بی‌صدای تمام خطاها):**

```Python
    try:
        save_to_database(user_data)
    except Exception:
        # فاجعه: چه دیتابیس قطع باشد، چه هارد پر شده باشد، چه دیتای یوزر غلط باشد،
        # سیستم بدون هیچ هشداری از روی آن رد می‌شود!
        pass 
```

- ✅ **رویکرد مهندسی (مدیریت دقیق و ثبت وقایع):**
```Python
    try:
        save_to_database(user_data)
    except ConnectionTimeoutError as e:
        # خطای شبکه مشخص است. آن را لاگ کرده و از مکانیزم تلاش مجدد (Retry) استفاده می‌کنیم.
        logger.error(f"DB Timeout for user {user_data.id}: {e}")
        schedule_retry(user_data)
```

## ۵. اعتبارسنجی در مرزها (Validation at the Boundaries)

**مفهوم معماری:** داده‌هایی که از بیرون وارد سیستم شما می‌شوند (ورودی‌های کاربر، پاسخ‌های یک API خارجی یا فایل‌های خوانده شده از هارد) موجوداتی "کثیف و غیرقابل اعتماد" در نظر گرفته می‌شوند.

**چرا اهمیت دارد؟** اگر دیتای کثیف اجازه پیدا کند در عمق لایه‌های سیستم (مکانیک‌ها و بیزینس لاجیک) نفوذ کند، شما مجبورید در ده‌ها کلاس مختلف اعتبارسنجی تکراری بنویسید.

**الزام اجرایی:**
داده‌ها باید دقیقاً در دروازه ورودی سیستم (لایه کنترلر، فرم‌های UI یا آداپتورهای شبکه) اعتبارسنجی و تصفیه (Sanitize) شوند. وقتی داده از مرز عبور کرد و به مدل‌های داخلی (Core Entities) تبدیل شد، بقیه سیستم می‌تواند با اطمینان ۱۰۰٪ روی تمیز و ایمن بودن آن حساب باز کند.
